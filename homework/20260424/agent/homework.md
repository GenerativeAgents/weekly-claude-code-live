## 調査メモ

`claude --agent <name>` は `.claude/agents/<name>.md` で定義したサブエージェントを
**メインスレッド全体の駆動役**として起動するフラグ。`@agent-<name>` によるタスク単位の
委譲と違い、セッション開始時点から終了まで、そのエージェントがメインとして動く。
frontmatter に書いたフィールド（`tools` / `model` / `permissionMode` 等）だけが
Claude Code 既定から上書きされる。

| 項目 | 通常の `claude` | `--agent <name>` |
| --- | --- | --- |
| システムプロンプト | Claude Code 既定 | エージェント定義ファイル本文 |
| 使えるツール | 全ツール | `tools:` frontmatter で絞り込み |
| モデル | CLI/設定で指定した値 | `model:` frontmatter が優先 |
| 初期コンテキスト | 空 | 定義ファイル本文が常時効いている |

出典: [Claude Code docs — agents.md](https://code.claude.com/docs/en/agents.md#invoke-subagents-explicitly)

## なぜ `--agent` を使うのか — ガードレールが本命

`--agent` の核心は **ツール allowlist がそのままプロセス境界のガードレール**になる点。
CLI 起動時に渡したツール集合がそのセッションで使える全部なので、会話の中で「ルール
無視して編集して」と押し返されても `Edit`/`Write` 自体が呼べず物理的に編集が起こらない。
システムプロンプトでの「読み取り専用です」というお願いとは強度が違う（後者は巧妙な
指示で揺らぐが、前者はプロセスにツールが無いので逃げ道がない）。

| ガード手段 | 強度 | 逃げ道 |
| --- | --- | --- |
| システムプロンプトでの注意書き | 弱い | 巧妙な指示で回避され得る |
| `tools:` allowlist (`--agent`) | 強い | プロセスにツールが無いので呼べない |
| `permissionMode: plan` | さらに強い | 書き込み系を持っていても plan で停止 |

主な効きどころ:

- **ヘッドレス CI / PR レビュー**: `claude --agent reviewer -p "..."` で読み取り専用
  ロボを安全に回す。
- **ロール別運用**: コードレビュー / アーキ調査 / セキュリティ確認など、責務ごとの
  エージェント定義をチームで持っておく。
- **プロンプトインジェクション耐性が必要な場面**: 外部入力を読ませるときに書き込み系
  ツールを最初から落としておけば、入力経由で誘導された変更が実行されない。

## 他の手段との使い分け

- **`--agent`（CLI フラグ）**: セッション丸ごと特定の役割に固定。境界が CLI 起動レイヤー
  なので強固。
- **`@agent-<name>`**: 通常セッション内で 1 タスクだけ別エージェントに任せる one-shot。
  境界は会話単位なので押し返しに弱い。
- **`/agents`**: エージェント一覧管理・定義編集。自動委譲もここ経由。
- **Task ツール（SDK から）**: 並列で独立コンテキストのサブエージェントを複数走らせたい
  とき。

一言まとめ: 「**プロセス起動レベルでガードを引きたい / 役割を固定したい**なら `--agent`、
**1 回こっきり**なら `@agent-<name>`、**スクリプト化**したいなら `--agent` + `-p`」。
